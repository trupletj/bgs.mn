# Feature Specification: Чөлөөний хүсэлт (Leave Requests)

**Feature Branch**: `001-leave-requests`

**Created**: 2026-05-26

**Status**: Draft

**Input**: User description: "Чөлөөний хүсэлт бүртгэх хэсгийг энд оруулах гэсэн юм. bgs-mobile-app дээр оруулж байсан бөгөөд Одоо энд хэрэгжүүлэх гэсэн юм. supabase дээр үндсэн table ууд нь аль хэдийн үүссэн байгаа бөгөөд илүү хүсэл илгээх болон хянах батлах хэсгүүдийг оруулах."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Ажилтан чөлөөний хүсэлт илгээнэ (Priority: P1)

Ажилтан вэб платформ дээрээс чөлөөний хүсэлт илгээнэ. Чөлөөний төрөл сонгож, эхлэх/дуусах огноо тэмдэглэж (шаардлагатай бол хагас өдөр гэж тэмдэглэж), шалтгаан болон нэмэлт баримт хавсаргаж илгээнэ. Илгээсний дараа хүсэлт автоматаар тогтоосон хяналтын процессийн эхний шатанд орж, тохирох эрхтэй хянагч нар томилогдоно.

**Why this priority**: Энэ бол гол хэрэглэгчийн үнэ цэн — ажилтан чөлөө авах ажиллагааг эхлүүлэх боломжтой болж байж л дараагийн чанарууд утга учиртай болно. P1 бол MVP-ийн үндэс.

**Independent Test**: Шинэ ажилтан логин хийж, "+ Чөлөөний хүсэлт" хуудас руу нэвтрэн, бүх шаардлагатай талбарыг бөглөж хүсэлт амжилттай илгээгдсэн төлвөөр харагдаж, дараа нь "Миний хүсэлтүүд" жагсаалтад орохыг шалгах боломжтой.

**Acceptance Scenarios**:

1. **Given** ажилтан логин хийсэн, **When** хүчинтэй талбартай (төрөл, эхлэх/дуусах огноо, тайлбар) хүсэлт илгээх, **Then** систем хүсэлтийг хадгалж, эхний шатны хянагч нарт даалгавар үүсгэж, амжилттай сэрэмжлүүлэг (toast) харуулна.
2. **Given** ажилтан хүсэлт илгээж эхэлсэн, **When** `end_date < start_date` оруулах, **Then** систем алдааны мэдэгдэл өгөх ба хүсэлтийг хадгалахгүй.
3. **Given** ажилтан хагас өдрийн чөлөө сонгосон, **When** хүсэлт илгээх, **Then** `duration_days` нь автомат тооцоологдож (жишээ нь 0.5) хадгалагдана.
4. **Given** ажилтан файл хавсаргасан, **When** хүсэлт илгээх, **Then** файл `leave-attachments` хадгалах bucket-руу байршиж, хүсэлттэй холбогдоно.

---

### User Story 2 — Хянагч (manager / HR) хүсэлтийг батлах буюу татгалзах (Priority: P1)

`leave:review` эрхтэй хэрэглэгч өөрт хуваарилагдсан хүсэлтүүдийг "Хяналт" хуудасны "Хүлээгдэж буй" tab-аас үзэж, тус бүрд нь Зөвшөөрөх эсвэл Татгалзах шийдвэр гаргана. Татгалзах үед заавал тайлбар оруулна. Тухайн шатны бүх шаардлагатай хянагч зөвшөөрсний дараа хүсэлт автоматаар дараагийн шат руу шилждэг ба дараагийн шатны хянагч нар томилогдоно. Бүх шат батлагдсаны дараа хүсэлтийн ерөнхий статус "Зөвшөөрөгдсөн" болно.

**Why this priority**: Хүсэлт илгээж байгаа эцсийн зорилго нь зөвшөөрөл авах учир хяналтын урсгал нь P1. Энэгүйгээр илгээсэн хүсэлтүүд нь тодорхой бус "pending"-д үлдэх юм.

**Independent Test**: Тестийн орчинд reviewer role-той хэрэглэгчээр логин хийж, "Чөлөө > Хяналт" badge-нд тоо гарч байгааг шалгаад, нэг хүсэлтэд орж зөвшөөрөх товч дарах. Шийдвэр амжилттай хадгалагдсан, badge тоо буурсан, хүсэлт "Хянагдсан" tab-руу шилжсэнийг шалгана.

**Acceptance Scenarios**:

1. **Given** хянагчид хуваарилагдсан хүлээгдэж буй хүсэлт байгаа, **When** хянагч sidebar-аас "Чөлөө" хэсэгт орох, **Then** "Хяналт" дэд цэс дээр хүлээгдэж буй хүсэлтийн тоо badge-аар харагдана.
2. **Given** хянагч хүсэлтийг нээсэн, **When** "Зөвшөөрөх" товч дарах, **Then** өөрийн reviewer мөр `approved` төлөвт орж, тухайн шатны бүх шаардлагатай хянагч зөвшөөрсөн бол шат шилжиж дараагийн хянагч нар томилогдоно.
3. **Given** хянагч хүсэлтийг нээсэн, **When** "Татгалзах" товч дарж тайлбаргүй илгээх, **Then** систем тайлбар шаардсан алдаа гаргана.
4. **Given** хянагч татгалзах тайлбар оруулсан, **When** "Татгалзах" товч дарах, **Then** хүсэлтийн ерөнхий статус `rejected` болж, цаашдын шатууд таслагдана.
5. **Given** хүсэлтийн бүх шат батлагдсан, **When** сүүлчийн хянагч зөвшөөрөх, **Then** хүсэлтийн ерөнхий статус `approved` болж, хүсэлт илгээгчид мэдэгдэнэ.

### User Story 3 — Ажилтан өөрийн хүсэлтийн түүх, статус харах (Priority: P2)

Ажилтан "Миний хүсэлтүүд" жагсаалтаас өөрийн илгээсэн бүх хүсэлтийн төлөв, шат, хэн хянаж байгааг хардаг. Идэвхтэй хүсэлтийг буцааж цуцлах боломжтой. Дэлгэрэнгүй хуудсанд статусын түүх (хэн, хэзээ, ямар тайлбартай) бүрэн харагдана.

**Why this priority**: Хүсэлт илгээсний дараа явц харах нь хүлээх үеийн тодорхой бус байдлыг арилгадаг — гэхдээ P1 user story-нуудаас үндэс нь хамаардаггүй учир P2.

**Independent Test**: Хүсэлт илгээсэн дараа "Миний хүсэлтүүд" хуудас руу орж, тухайн хүсэлт жагсаалтад харагдаж, дэлгэрэнгүй хуудсанд одоогийн шат + хянагч нарын статус харагдаж, "Цуцлах" товч дарж амжилттай цуцлагдсаныг шалгах.

**Acceptance Scenarios**:

1. **Given** ажилтан өмнө нь хүсэлт илгээсэн, **When** "Миний хүсэлтүүд" хуудсыг нээх, **Then** хүсэлт бүрд статус, чөлөөний төрөл, огноо, үргэлжлэх хугацаа харагдана.
2. **Given** ажилтан жагсаалтаас идэвхтэй (`in_review`) хүсэлтээ нээх, **When** "Цуцлах" товч дарах, **Then** хүсэлтийн статус `cancelled` болж, идэвхтэй reviewer task-ууд хаагдана.
3. **Given** ажилтан жагсаалтад статусаар filter хийх, **When** "Зөвшөөрөгдсөн" filter сонгох, **Then** зөвхөн `approved` статустай хүсэлтүүд харагдана.

---

### User Story 4 — Admin чөлөөний төрөл, workflow тохиргоог удирдана (Priority: P3)

`leave:admin` буюу `super_admin` эрхтэй хэрэглэгч чөлөөний төрөл (Leave Type) болон тэдгээрт харгалзах хяналтын процесс (Process / Step / Step Role) үүсгэх, засах, идэвхгүй болгох боломжтой. Чөлөөний төрөл бүрд өөр өөр шат, шат бүрд өөр өөр role-тай хянагч нарыг сонгож тохируулна.

**Why this priority**: Бизнесийн дүрэм цаг хугацаанд өөрчлөгдөж болох ч эхний release-д шаардлагатай бол төрөл/процессыг гар аргаар Supabase Studio-аас орж тохируулсан ч болно. UI нь админуудад тав тух өгөх P3 өргөтгөл.

**Independent Test**: Super_admin хэрэглэгчээр логин хийж, "Чөлөөний төрөл" хуудсанд орж шинэ төрөл үүсгэх, тэр төрөлд 2-шаттай процесс тохируулах, дараа нь P1-ийн дагуу хүсэлт илгээх — шинэ хүсэлт нь шинэ процессийг ашиглаж эхний шатны хянагч нарт томилогдсон эсэхийг шалгах.

**Acceptance Scenarios**:

1. **Given** super_admin логин хийсэн, **When** "Чөлөөний төрөл" хуудаснаас шинэ төрөл үүсгэх, **Then** төрөл идэвхтэй болж ажилтны хүсэлт илгээх форм дээр сонгож болохуйцаар харагдана.
2. **Given** super_admin хяналтын процесс үүсгэсэн, **When** тухайн процесст 2 шат, шат тус бүрд role оноох, **Then** уг процессыг ашигладаг төрлийн хүсэлт илгээх үед эхний шатны role-тай ажилтан reviewer болон автоматаар томилогдоно.

---

### Edge Cases

- Хэрэглэгч өнгөрсөн огноогоор хүсэлт илгээх (жишээ нь өвдсөн өдөртөө буцаан) — зөвшөөрөх боловч "Эргэн харгалзах хүсэлт" хэмээн тэмдэглэх.
- `end_date < start_date` оруулсан тохиолдолд форм-ын валидаци алдаа гаргана.
- Нэг ажилтан давхцсан огноотой хоёр хүсэлт илгээх — анхааруулга үзүүлэх (хорихгүй) ба HR-д зориулсан "давхцал" дохио харуулна.
- Workflow process нь дундуур идэвхгүй болсон тохиолдолд урьд үүссэн `in_review` хүсэлтүүд хуучин процессоо дагаж дуусгана (process snapshot нь instance-д тэмдэглэгдсэн байх).
- Хүсэлт илгээгч нь өөртэйгөө хянагч болох тохиолдол (жишээ нь HR ажилтан өөрийн чөлөө илгээх) — өөрийн мөрийг автомат `skipped` болгож, бусад хянагч нараар л баталгаажуулна.
- Файл хавсралтын хэмжээ хязгаар: нэг файл 10 МБ-аас ихгүй, зөвшөөрөгдсөн төрөл (PDF, JPG, PNG, DOCX).
- Reviewer-ын role нь хүсэлт батлагдахаас өмнө хасагдсан тохиолдолд тухайн reviewer мөр `skipped` болж, бусад мөрийн дагуу шат үргэлжилнэ.
- Хүсэлтийг илгээгч нь идэвхгүй болсон (ажилтан гарсан) тохиолдол — хүсэлт автоматаар цуцлагдана.

## Requirements *(mandatory)*

### Functional Requirements

**Submitting requests (P1)**
- **FR-001**: Ажилтан чөлөөний төрөл, эхлэх огноо (`start_date`), дуусах огноо (`end_date`), хагас өдрийн тэмдэг (`is_half_day`), тайлбар, нэмэлт файл хавсралттай хүсэлт илгээх боломжтой байна.
- **FR-002**: Систем `start_date <= end_date` шалгана. `end_date` нь өнгөрсөн огноо байж болох (эргэн харгалзах хүсэлт), гэхдээ форм дээр анхааруулга харуулна.
- **FR-003**: Систем `duration_days` талбарыг `start_date`, `end_date`, `is_half_day` талбараас автоматаар тооцоолж хадгална.
- **FR-004**: Хүсэлт амжилттай үүсэх үед сонгосон чөлөөний төрөлд тохирох workflow process-ын Step 1-д шаардлагатай role-той бүх ажилтан reviewer болж томилогдоно.
- **FR-013**: Файл хавсралт нь Supabase Storage `leave-attachments` bucket-руу хадгалагдаж, хүсэлттэй холбогдоно. Нэг файл 10 МБ-аас хэтрэхгүй, PDF / JPG / PNG / DOCX төрлөөр хязгаарлагдана.

**Review and approval (P1)**
- **FR-005**: Reviewer "Зөвшөөрөх" буюу "Татгалзах" товчоор шийдвэрээ оруулна. "Татгалзах" сонголтод тайлбар (`note`) заавал шаардлагатай.
- **FR-006**: Шатанд хуваарилагдсан аль ч reviewer татгалзвал хүсэлтийн ерөнхий статус `rejected` болж, цаашдын шатны reviewer мөрүүд `skipped` болно.
- **FR-007**: Тухайн шатны бүх шаардлагатай reviewer зөвшөөрсний дараа `order_step_reviewers`-той ижил логикийн дагуу хүсэлт автоматаар дараагийн шат руу шилжиж, тэр шатны хянагч нарыг роль-аар нь автомат томилно.
- **FR-008**: Бүх шат бүрэн зөвшөөрөгдсөн үед хүсэлтийн ерөнхий статус `approved` болно.
- **FR-010**: Хянагчийн "хүлээгдэж буй" reviewer task-ын тоо sidebar nav-д "Чөлөө > Хяналт" дэд цэс дээр badge-аар харагдана. Тоо нь зөвхөн тухайн хэрэглэгчид `pending` мөр байгаа эсэхийг тоолж гаргана.

**User-facing visibility (P2)**
- **FR-009**: Ажилтан өөрийн илгээсэн хүсэлтийн жагсаалт, дэлгэрэнгүй (workflow steps, шийдвэрийн түүх) -г үзэх ба идэвхтэй (`in_review`) хүсэлтийг `cancelled` болгох эрхтэй.

**Workflow administration (P3)**
- **FR-014**: `super_admin` болон `leave:admin` эрхтэй хэрэглэгч `leave_types` хүснэгтийг CRUD, тус бүрд харгалзах workflow process болон шат / role-уудыг тохируулна.

**Security and access**
- **FR-011**: Эрхийн систем (RBAC) дараах permission-ыг танина: `leave:access` (өөрийн хүсэлт харах), `leave:create` (хүсэлт илгээх), `leave:review` (хяналт хийх), `leave:admin` (чөлөөний төрөл болон процесс удирдах).
- **FR-012**: Row Level Security (RLS) policy-ийн дагуу: ажилтан зөвхөн өөрийн хүсэлтийг хардаг, reviewer нь өөрт хуваарилагдсан хүсэлтийг хардаг, `leave:admin` эрхтэй хэрэглэгч бүх хүсэлтийг хардаг.

**Audit and history**
- **FR-015**: Хүсэлтийн ерөнхий статус өөрчлөгдөх бүрд (`pending → in_review`, `in_review → approved | rejected | cancelled`) системд бүртгэгдэх ба дэлгэрэнгүй хуудаснаас үзэх боломжтой байна. Reviewer тус бүрийн шийдвэр (`approved/rejected/skipped` + тайлбар + огноо) мөн хадгалагдана.

**Data continuity**
- **FR-016**: Mobile апп-аас одоо хүртэл илгээгдсэн флат `pending/approved/rejected` статустай хүсэлтүүд өөрчлөгдөхгүй харагдсаар байх ба шинэ workflow-той хүсэлтүүдтэй зэрэгцэн ажиллана. Шаардлагатай бол өгөгдөл шилжүүлэх migration ажиллагаа хийнэ.

### Key Entities

- **LeaveType (`leave_types`)** — Чөлөөний категори. Дүрэм: `id`, `name`, `is_active`, харгалзах workflow process-той холбогдоно (`process_id` FK эсвэл `leave_request_processes.leave_type_id` joins).
- **LeaveRequest (`leave_requests`)** — Үндсэн хүсэлт. Шинэ багана нэмэх шаардлагатай: `start_date`, `end_date`, `is_half_day`, `updated_at`. Статус CHECK-д `in_review`, `cancelled` нэмэгдэнэ.
- **LeaveRequestProcess** *(шинэ)* — Workflow template (нэр, тайлбар, тухайн чөлөөний төрөлтэй холбогдсон, идэвхтэй эсэх).
- **LeaveRequestStep** *(шинэ)* — Процессийн доторх дараалсан шат: `process_id`, `step_order`, `name`.
- **LeaveRequestStepRole** *(шинэ)* — Шатад шаардлагатай role-уудын холбоо: `step_id`, `role_id`.
- **LeaveRequestInstance** *(шинэ)* — Running workflow state: `leave_request_id`, `process_id`, `current_step_order`, `status`.
- **LeaveRequestStepReviewer** *(шинэ)* — Reviewer бүрд нэг мөр: `instance_id`, `step_order`, `reviewer_profile_id`, `status (pending|approved|rejected|skipped)`, `note`, `reviewed_at`.
- **LeaveRequestStatusHistory** *(шинэ — audit log)* — Хүсэлтийн ерөнхий статусын шилжилт бүрийг бүртгэх: `leave_request_id`, `from_status`, `to_status`, `changed_by_profile_id`, `note`, `changed_at`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Ажилтан шинэ чөлөөний хүсэлтийг 60 секундын дотор бөглөж илгээж дуусна (форм нээгдсэнээс эхлэн).
- **SC-002**: Хянагч хүсэлт нээж шийдвэр (зөвшөөрөх буюу татгалзах + тайлбар) оруулах нь 30 секундын дотор хийгдэнэ.
- **SC-003**: Хүсэлтийн статусын өөрчлөлт болон шат бүрийн reviewer шийдвэр 100 хувь audit log-д бүртгэгдэнэ (алдсан өөрчлөлт байхгүй).
- **SC-004**: Ажилтан өөрийн идэвхтэй хүсэлт ямар шатанд, хэн дээр хүлээгдэж байгааг нэг л дэлгэцээс харна (нэмэлт навигаци хэрэггүй).
- **SC-005**: Хянагчид хуваарилагдсан хүлээгдэж буй хүсэлтийн тоо sidebar дээр real-time-д ойролцоо (≤30 секундын зөрөөтэй) шинэчлэгдэнэ.
- **SC-006**: HR-ын баримтжуулсан "чөлөөний шийдвэр хүлээх" дундаж хугацаа цаасан/мэйл-аар явсан үетэй харьцуулахад 50 хувиар буурна (release дараа 30 хоногийн хугацаанд хэмжих).

## Assumptions

- Mobile апп-д одоо ашиглаж буй `leave_types` болон `leave_requests` хүснэгтийг өргөтгөж, өгөгдөл алдалгүй migration-аар workflow багана нэмэх боломжтой гэж үзэв.
- Workflow тохиргоо нь чөлөөний төрөл тус бүрд 1 идэвхтэй process-той байх ба нэгэн зэрэг олон идэвхтэй process авч буй тохиолдол одоогийн scope-ноос гадуур.
- v1-д чөлөөний жилийн квот (leave balance) автомат хянахгүй. HR гараар хянаж, цаашид нэмж болно.
- RBAC-ийн `permissions` хүснэгтэд `leave` модулийн permission-уудыг шинээр нэмж бүртгэх боломжтой (одоо байгаа `order`, `dining` зэрэгтэй ижил загвараар).
- Supabase Storage `leave-attachments` bucket аль хэдийн үүссэн (mobile-аас ашигладаг).
- Хянагч / админ-ын role нэр нь одоо байгаа `hr_emp`, `super_admin` зэргийг ашиглах ба шаардлагатай бол `leave_reviewer` зэрэг нэмэлт role нэмэгдэнэ — нарийвчлал planning шатанд тодорхой болно.
- Шинэ хүсэлт илгээх үед mobile апп нь шууд workflow-той тохиргоог дагах боломжтой эсвэл flat-status mode-оор үргэлжилнэ. Cross-platform нийцтэй байдал planning шатанд тодорхой болно.

## Dependencies

- Одоо ажиллаж буй Supabase `leave_types`, `leave_requests` хүснэгт болон `leave-attachments` Storage bucket.
- Одоогийн орчны `permissions` / `role_permissions` бүтэц (`actions/rbac.ts` дотор `hasPermission` ашигладаг).
- Sidebar nav badge суурь бүтэц (`actions/nav.ts` болон `getPendingOrderReviewCountForCurrentUser` загвар).
- Orders module-аас reused UI/logic (workflow indicator, review queue, detail layout).
